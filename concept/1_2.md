# Vii v1.2 — "Code Anything" Roadmap

> **Philosophy guardrail**: Every new word must unlock an entire category of programs that were previously impossible or impractical in vii. If a feature can be built as a `.vii` library, it should be. New keywords are the last resort.

---

## Feature Checklist

### Core Language (C changes)

- [x] **Unary minus** — `x = -5`, `y = -x`, `z = -(a + b)` — math expressions read naturally
- [x] **String comparison** — `<`, `>`, `<=`, `>=` work on strings via `strcmp` — sorting, ordering, binary search
- [x] **`break`** — early loop exit — search algorithms, parsers, state machines
- [x] **`slice`** — substring/sublist extraction — parsers, tokenizers, text processing
- [x] **`type`** — type introspection — polymorphic functions, safe input handling
- [x] **`time`** — epoch timestamp — benchmarks, scheduling, games, unique seeds
- [x] **`append`** mode for `put` — log files, data pipelines
- [x] **`dict` + `key`** — key-value maps — JSON, databases, caches, configs, graphs, frequency counts
- [x] **`at` on dicts** — `d at "name"` — dict value lookup
- [x] **`len` on dicts** — key count
- [x] **Parens for grouping** — `(a + b) * c`, `type (1 == 1)`, `tostr (x > 0)` — eliminates intermediate variable workarounds
- [x] **Compile-time `IF`/`ELSE`** — cross-platform code, ALL_CAPS constants — OS-specific builds without runtime cost
- [x] **`-D` CLI flag** — compile-time defines for `IF` macros

## Candidate Features for v1.3

Each must unlock an **entire category of programs** currently impossible or impractical.

### 1. `split` — String Splitting (+1 keyword)

**Unlocks**: CSV parsers, word counters, tokenizers, log analyzers, shell output parsers

**Why**: Today, splitting a string by a delimiter requires a manual loop with `at`, `ord`, and `slice`. This is the #1 string operation missing. Every real program that reads files or user input needs to split.

```vii
line = "one,two,three"
parts = split line ","       # ["one", "two", "three"]

csv = "name,age"
split csv ","                # ["name", "age"]
```

**Keyword count**: +1 (`split`)

### 2. `trim` — Whitespace Stripping (+1 keyword)

**Unlocks**: Clean input handling, file parsing, user input sanitization

**Why**: Every program that reads from `ask` or files gets trailing whitespace. Currently requires a manual loop. `trim` is the companion to `split` — together they handle 90% of string processing.

```vii
s = "  hello  "
trim s                       # "hello"
```

**Keyword count**: +1 (`trim`)

### 3. `replace` — String Replacement (+1 keyword)

**Unlocks**: Template engines, config generators, HTML sanitizers, text transformers

**Why**: Without `replace`, you can't build a template system, sanitize input, or transform text formats. It's the third essential string operation alongside `slice` and `split`.

```vii
s = "Hello World"
replace s "World" "Vii"      # "Hello Vii"
```

**Keyword count**: +1 (`replace`)

### 4. `for` — List Iteration (+1 keyword)

**Unlocks**: Readable data processing, map/filter patterns, beginner ergonomics

**Why**: `i = 0; while i < len lst; ... i = i + 1` is the most common loop pattern. It's 3 lines of boilerplate for what should be 1. `for` doesn't add new capability — it removes friction on the most common operation.

```vii
for item in items
  item                        # print each item

for name in names
  "Hello " + name
```

**Keyword count**: +1 (`for`, `in` — but `in` could be a soft keyword recognized only in `for` context)

### 5. `ref` / `->ptr` — Pass by Reference (+1 keyword, +1 type)

**Unlocks**: In-place mutation of large objects, swap patterns, efficient function params

**Why**: Currently all values are copied when passed to functions. For large lists/dicts this is wasteful. `ref` creates a transparent alias — reads and writes go to the original.

```vii
x = 10
y ptr = x                   # y is an alias for x
y = 20                      # x is now 20 too

do func y->ptr              # y is passed by reference
  y = 20                    # modifies the original
```

**Keyword count**: +1 (`ptr` as type annotation with `->ptr`)

### 6. `try` / `safe` — Error Recovery (+1 keyword)

**Unlocks**: Long-running servers, robust CLI tools, programs that don't crash on bad input

**Why**: Currently any error calls `exit(1)`. You can't write a server or interactive tool that survives bad input. `safe` wraps an expression and returns `none` on error instead of crashing.

```vii
result = safe ask "missing.txt"    # returns "" instead of crashing
if type result == "none"
  "File not found, using defaults"
```

**Keyword count**: +1 (`safe`) — no `catch` needed, just returns `none` on error

## What Vii Does NOT Add (Scope Creep Kill List)

These are deliberately excluded to preserve vii's minimalist identity:

- **Classes/OOP** — Use dicts + functions. Vii doesn't need `class`.
- **Exception handling** — Use `type` checks and `if` guards. `try/catch` adds 2 keywords for marginal gain.
- **Concurrency/threads** — Use `sys` to spawn processes. Threading is an OS concern.
- **First-class functions** — Functions are already callable by name. Passing them as values would require a `VAL_FUNC` type and breaks the "no punctuation" rule (callbacks need parens).
- **Pattern matching** — Use `if`/`else if` chains with `at` and `type`.
- **Modules/namespaces** — `paste` is sufficient. Taco handles distribution.
- **Operator precedence** — Left-to-right is simpler. Use parens for grouping.
- **Closures** — Functions already access parent scope via `Table.parent`.
- **Regex** — Build it in `lib/regex.vii` using `ord`/`chr`/`at` if needed. Not a language feature.
- **Float formatting** — Build `lib/fmt.vii` if needed. Not a language feature.
